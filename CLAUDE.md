## What this is

winmux is a native tmux alternative for Windows, written in Rust on top of ConPTY. It runs *inside* the user's terminal (like tmux does) and draws panes/borders/status bar with VT escape sequences — it is not its own GUI window. Guiding principle: **be exactly like tmux** — wherever a design choice exists, match tmux's real defaults so tmux users are immediately at home.

Current state: all four planned sub-projects are complete — **the planned project scope is done** — plus a second parity-hardening wave (SP6) and a third (SP7) beyond the original scope.

Multiple sessions and windows are supported, along with a tmux CLI subset: `new-session`/`new`, `attach-session`/`attach`/`a`, `detach-client`, `list-sessions`/`ls`, `list-windows`/`lsw`, `has-session`/`has`, `kill-session`, `kill-server`, `rename-session`, `rename-window`, plus global `-L <socket-name>`. ## Commands

`cargo` may not be on PATH in fresh shells — it lives at `C:\Users\poon\.cargo\bin`:

```bash
export PATH="$HOME/.cargo/bin:$PATH"        # Bash
$env:Path = "$env:USERPROFILE\.cargo\bin;$env:Path"   # PowerShell
```

```bash
cargo build                                  # debug build
cargo test                                   # everything: unit + integration + e2e
cargo test layout::                          # one module's unit tests (also grid::, render::, input::, model::, status::, protocol::, pipe::)
cargo test --test pty_smoke                  # ConPTY integration tests (spawn real cmd.exe/powershell.exe; SP7 adds a permanent regression pin for the mouse-DECSET-not-relayed platform gotcha, see below)
cargo test --test pipe_smoke                 # named-pipe transport integration tests
cargo test --test server_proto               # headless server protocol tests — no console, no ConPTY host; speaks raw frames over the pipe
cargo test --test e2e                        # sub-project 1 style e2e: drives winmux.exe under a ConPTY (single session)
cargo test --test e2e_sessions               # sub-project 2 e2e: detach/reattach, windows, kill-session over real named pipes
cargo test --test e2e_config                 # sub-project 3 e2e (+ SP6, SP7): .tmux.conf round trip via -f, prefix-: command prompt, send-keys CLI, real user .tmux.conf loads clean via -f; SP7 adds a closeout config exercising window-scoped setw, a #{?...}/#{==:...} conditional format, a config bind on a mouse pseudo-key, and set-clipboard
cargo test --test e2e_copy_mouse             # sub-project 4 e2e (+ SP6): copy-mode roundtrip (scroll into history, select, copy, paste), mouse click-focus, mouse-wheel scroll into copy mode, mouse drag-select-and-release copy
cargo test -- --test-threads=1               # serialize when diagnosing e2e flakiness (each e2e test uses a unique -L socket, so parallel `cargo test` is safe by construction, but real ConPTY/process load under full parallelism can still make timing assertions flaky)
cargo clippy --all-targets -- -D warnings    # lint gate — must stay clean
cargo build --release                        # binary at target\release\winmux.exe
cargo test host::tests -- --ignored          # manual host smoke test (needs a real console)
```

Windows-only: the integration/e2e tests spawn real processes through ConPTY and bind real named pipes; require Win10 1809+. A running `winmux.exe` **or a running background server** locks `target\release\winmux.exe` — exit any attached client and run `winmux -L <socket> kill-server` (or `target\release\winmux.exe kill-server` if not on PATH) before rebuilding release.

`tests/server_proto.rs` (229 tests as of SP7, headless protocol-level) is fast but spins up a real named-pipe server per test; under `cargo test`'s default full-parallelism it has occasionally shown timing flakiness on a loaded machine (each test still uses its own isolated server/socket, so failures are timing, not cross-test interference). If it flakes, retry with a capped thread count rather than full serialization: `cargo test --test server_proto -- --test-threads=4`. SP7's test-debt batch (Task 18) already widened the two previously-flakiest tests' timing margins (`recv_output_settled`, closes follow-up #58) and made `drain_after_exit` condition-based rather than a fixed sleep (follow-up #12); a residual, narrower flake class remains around `source_file_runtime` under full parallelism (documented, out of scope — same class as the general note above).

## Architecture

Core insight: **every pane is its own tiny terminal emulator.** ConPTY hands us the raw VT output stream of a shell; `grid` parses it into a cell matrix; `render` stitches all pane grids + borders + status bar into one frame per client, diffed against that client's previous frame; only changed cells are written out.

One binary, `winmux.exe`, plays three roles selected by argv (`src/cli.rs` parses this):

| Invocation | Role |
|---|---|
| `winmux [flags] [command ...]` | CLI: parse args, connect to the pipe (auto-starting the server only for commands that create state, e.g. `new-session`), then either run a one-shot control command or become an attached client. |
| *(same binary, becomes the attached client)* | Client: thin — forwards raw stdin, writes server-composed VT bytes to stdout verbatim, draws nothing itself. |
| `winmux __server --pipe <full-pipe-name>` | Hidden: the server event loop. Spawned detached (`DETACHED_PROCESS \| CREATE_NEW_PROCESS_GROUP`) so it survives the launching console closing. |

Server-side data flow (all state owned by the server's main thread — no locks on core state):

```
pane reader/waiter threads ─┐
client pipe reader threads ─┼→ mpsc<ServerEvent> → server main loop (owns ALL state) → per-client writer thread ← mpsc<Vec<u8>>
accept thread ──────────────┘
```

The main loop owns the session/window `Registry`, every pane's `Pty` + `Grid` (dropped from the map once the pane exits), and every attached client's `Renderer` + `InputMachine` + prompt/confirm state. Each client has its own prefix-key state machine, so multiple clients can be mid-prefix independently. A `recv_timeout(50ms)` tick refreshes the clock and coalesces events (see follow-up #4); resize is client-pushed (`Resize` frames), not polled.

### The locked interface contract

Public APIs of every module are governed by four contract files: `docs/specs/2026-07-06-mvp-interfaces.md` (geom/layout/grid/render/pty/host), `docs/specs/2026-07-07-server-client-interfaces.md` (protocol/pipe/model/status/server/cli/client), `docs/specs/2026-07-07-command-config-interfaces.md` (keys/style/cmd/options/bindings/server::dispatch), and `docs/specs/2026-07-07-parity-polish-interfaces.md` (sub-project 4's copy-mode/mouse/layout-preset/overlay additions to the same modules). Neither SP6 (tmux parity wave 2) nor SP7 (tmux parity wave 3) introduced a fifth contract file — every SP6/SP7 change amended one or more of these four in the same commit as its code change, including SP7's brand-new `format` module (folded into the `command-config` file as a new section rather than getting its own file). Do not add public surface or change signatures ad hoc: if a signature must change, update the relevant contract file(s) and every consumer in the same commit. Private helpers are unconstrained.

### Non-negotiable invariant: never wreck the user's terminal

Every exit path (normal exit, error, panic) must restore console modes, code pages, alt-screen, cursor, and SGR. This is implemented as an idempotent pair in `src/host.rs`: `Host`'s `Drop` plus a global panic hook, both reading a shared `RESTORE` snapshot (poison-recovering mutex; published before the first mutation, follow-up #3). If you touch anything `Host::enter` mutates, you must also restore it in `apply_restore`. The server process is headless (no console to restore) and instead installs a panic hook that logs to `%LOCALAPPDATA%\winmux\server.log`.

## Hard-won platform gotchas (do not regress)

- **ConPTY + redirected stdio** (`src/pty.rs`): when the *parent's* std handles are not a console (e.g. under `cargo test`), `CreateProcessW` leaks them into the child past the pseudoconsole, so output bypasses the pipe. Fix is `STARTF_USESTDHANDLES` with null handles — applied *only* when `parent_stdio_is_redirected()`; setting it unconditionally is unnecessary interactively.
- **Console code pages** (`src/host.rs`): output must be forced to CP_UTF8 or box-drawing borders mangle under OEM code pages — and the original code pages must be saved and restored on exit.
- **PSModulePath** (`src/main.rs`): PowerShell 7 exports a `PSModulePath` that makes `powershell.exe` 5.1 panes resolve PSReadLine to PS7's script module, which execution policy blocks ("Cannot load PSReadline module"). winmux clears the variable at startup **in every role** (including the server, since it is the pane parent now) before any pane spawns; panes rebuild their own default.
- **ConPTY exit protocol** (`src/pty.rs` / `src/server.rs`): the output pipe does not reliably EOF when the child exits. A waiter thread per pane does `WaitForSingleObject` on the process handle and sends an exit event; dropping the `Pty` (TerminateProcess → ClosePseudoConsole → CloseHandle, in that order) is what unblocks a reader stuck in `ReadFile`.
- **Zero-size geometry**: `layout::rects()` may return zero-size rects when the terminal is tiny; `grid` clamps to 1x1; every consumer must tolerate w==0/h==0 rects rather than assume drawability.
- **Overlapped named-pipe I/O is mandatory, not cosmetic** (`src/pipe.rs`): every pipe handle is opened with `FILE_FLAG_OVERLAPPED`, and `Read`/`Write` issue raw `ReadFile`/`WriteFile` against a per-connection event instead of going through `File`'s synchronous path. Without this, a synchronous handle serializes I/O across its duplicated clones — a pending read on one clone would block a concurrent write on another, which the reader-thread/writer-thread split requires to work.
- **`FILE_FLAG_FIRST_PIPE_INSTANCE` prevents split-brain autostart** (`src/pipe.rs`): only the *initial* `CreateNamedPipeW` for a socket name sets this flag, so it fails with `ERROR_ACCESS_DENIED` if another process already owns the name — the loser of a double cold-start autostart race then exits instead of silently joining the winner's instance pool as a second, round-robined server on the same pipe name. Subsequent accept-time instances (added by the same server) must NOT set the flag.
- **ConPTY does not relay a hosted process's mouse-mode DECSET** (found in SP7 Task 19 closeout): a `CSI ?1000h`-class private-mode sequence written by a real pane app never reaches the reader of the pseudoconsole's output pipe at all (confirmed empirically — an ordinary SGR sequence sent the same way survives byte-for-byte in the same capture, ruling out "the reader/harness doesn't work"; regression-pinned by `tests/pty_smoke.rs::mouse_decset_private_mode_is_not_relayed_by_conpty`). This means `Grid::mouse_proto` can never observe a real hosted app's own mouse-mode request over ConPTY on this platform, and retroactively explains three previously-separate "couldn't build this e2e test" notes across SP4-SP7 (see `docs/follow-ups.md` #84 for the full writeup) — a platform ceiling, not a winmux defect.

## Keybindings (tmux defaults — table-driven and rebindable since sub-project 3)

These are the *default* bindings loaded into `src/bindings.rs`'s `Bindings::default()`; any of them (including the prefix key itself) can be changed at runtime (`bind-key`/`unbind-key`, or the `prefix-:` command prompt) or from a config file (`set -g prefix <key>`, `bind <key> <command...>`, `unbind <key>`). Prefix defaults to `Ctrl-b`, then:

- Panes (unchanged since the MVP): `%` split left/right · `"` split top/bottom · arrows focus · `o` cycle · `;` last pane · `x` kill (y/n confirm) · `z` zoom · `Ctrl-arrow` resize (repeatable, 500ms window) · the prefix key again sends it literally.
- Windows/sessions (since sub-project 2): `c` new window (create + switch) · `n`/`p` next/previous window (wraps) · `l` toggle to last window · `0`-`9` select window by index · `&` kill window (confirm prompt `kill-window <name>? (y/n)`) · `,` rename window (status-line prompt, pre-filled) · `$` rename session (prompt, pre-filled) · `d` detach (`[detached (from session x)]`, client exits 0) · `(`/`)` switch this client to the previous/next session.
- Command layer (sub-project 3): `:` opens the status-line `prefix-:` command prompt — anything from the CLI subset (`rename-window foo`, `split-window -h`, `set -g status off`, ...) can be typed and executed there, the same dispatcher a keybinding or the `winmux <cmd>` CLI goes through.
- Copy mode (sub-project 4; `C-k`/`M-m` added in SP7): `[` enters copy mode on the focused pane; `PPage` enters pre-scrolled one page up. `mode-keys` (`emacs` default, or `vi`) selects which key table is active while in copy mode — arrows/`C-b`/`C-n`/`C-p`/`C-f`/`C-a`/`C-e`/`Home`/`End`/`M-<`/`M->`/`PPage` page up and `NPage`/Space page down in the emacs table; `hjkl`+arrows/`w`/`b`/`e`/`0`/`$`/`g`/`G`/`H`/`M`/`L`/`K`/`J`/`C-u`/`C-d`/`C-b`/`C-f` in the vi table. Both tables share: `q` cancel (Escape also cancels in the emacs table, but in vi it clears the selection instead — `q` is the only vi-table cancel key), `C-Space` (emacs) or Space (vi) begin selection, `R`/`v` rectangle-toggle, `o` other-end, `C-w`/`M-w` (emacs) or Enter (vi) copy-selection-and-cancel, `C-g` (emacs) clear-selection, `C-s`/`/` search-forward, `C-r`/`?` search-backward, `n`/`N` repeat/reverse search. Emacs-only (SP7, closes follow-up #56): `C-k` copy-end-of-line-and-cancel, `M-m` back-to-indentation (winmux's own naming choice — plain copy, no pipe, unlike tmux master's `copy-pipe-end-of-line-and-cancel`; see `CopyAction::EndOfLineAndCancel`'s doc comment). `copy-pipe`/`copy-pipe-and-cancel [command]` (SP7, closes #53/#55) have no default binding (real tmux's don't either, since they need a command argument) but work via `bind`/the `:` prompt; every copy destination also emits OSC 52 to the client when `set-clipboard` (`on`/`external`, SP7) says to. The `[scroll/history]` position indicator is drawn top-right of the pane while in copy mode.
- Paste buffers (sub-project 4): `]` paste the newest buffer (`paste-buffer -p`) · `#` show a one-line buffer-list summary (`list-buffers`) · `-` delete the newest buffer. `paste-buffer -p`'s bracketed-paste wrap (SP7, closes #53) only actually wraps when the TARGET pane has itself requested bracketed-paste mode (`DECSET 2004`).
- Layouts (sub-project 4): `Space` cycle through the five layout presets (`next-layout`) · `M-1`..`M-5` select a layout by name (even-horizontal/even-vertical/main-horizontal/main-vertical/tiled) · `{`/`}` `swap-pane -U`/`-D` · `C-o`/`M-o` `rotate-window` (forward/`-D`). `swap-pane -s`/`-t` (SP7, closes #41/#42) now supports real cross-window/session swaps and `-U`/`-D` relative to an explicit `-s`.
- Window ops (sub-project 4; cross-window/session ops added in SP7): `!` `break-pane` (focused pane out into a new window; SP7 adds `-s src`/`-t [session:]dst`, closing #44) · `.` prompt → `move-window -t <index>` (SP7's `-t session:index` form can relocate a window into a DIFFERENT session, closing #45) · `f` prompt → `find-window <pattern>` — **changed in SP7** (closes #46): now ALWAYS opens the `choose-tree` overlay filtered to the matches, even for exactly one match (previously jumped directly to the first match) — matches real tmux's `window-tree` sugar exactly; supports `-C`/`-N`/`-T`/`-i`/`-r` (real regex, via the new `regex` crate dependency) · `'` prompt → `select-window -t :<index>`.
- Overlays (sub-project 4; tree view + preview since SP6; choosers/tagging/sort/filter since SP7): `w` `choose-tree -w` (this session's windows) · `s` `choose-tree -s` (all sessions, sessions collapsed by default) · `=` `choose-buffer` (SP7, closes #48) · `D` `choose-client` (SP7, closes #49) — all four views share the same generalized overlay machinery (SP7): Up/Down/`j`/`k` move (default selection = the current item), Enter switches/pastes/detaches per view, `x` kills/deletes selected (confirm for sessions/windows only) — or, if any rows are tagged, applies to every TAGGED row instead (SP7, closes #50) — `q`/Escape/`C-c` cancel, `Left`/`Right` collapse/expand a session node (SP6, choose-tree only), `v` cycles the live preview box off → BIG → normal (SP6, tmux sizing/box chrome, choose-tree only); SP7 tagging/sort/filter (closes #50): `t` toggle a tag (`*` marker), `T` untag all, `C-t` tag every visible root row, `O` cycle sort field (per-view choices), `r` reverse sort, `f` open a filter prompt (substring, case-insensitive), `c` clear the filter · `q` `display-panes` (big digit overlay for `display-panes-time` ms or until a digit/other key; a digit key jumps to that pane).
- Clock mode (SP6): `t` enters `clock-mode` on the focused pane (a big digital clock, styled by `clock-mode-colour`/`clock-mode-style`); any key or mouse event exits.
- Mouse (sub-project 4, off by default — `set -g mouse on`; drag/border semantics refined in SP6; table-driven + app passthrough since SP7): click focuses a pane; dragging a pane border live-resizes the split in ALL FOUR directions (leftward/upward drags were a silent no-op pre-SP6, follow-up #66, now fixed); click/wheel on a status-row window tab selects/cycles windows; wheel-up over a live pane enters copy mode scrolled 5 lines, wheel in copy mode scrolls 5 lines; a click-drag on a LIVE (non-copy-mode) pane now auto-enters copy mode on the first `Drag` event and starts a selection there too (SP6 Task 6 — previously drag-to-select only worked if the pane was already in copy mode); release copies the selection against the pane under the pointer AT RELEASE (SP6 Task 6/7) and exits back to Normal; drag autoscrolls at pane edges (SP6 Task 7); double/triple-click select+copy a word/line, extendable by a further drag (SP6 Task 7, 3-class word model, `word-separators` option); right-click on a pane/window-tab/session-name opens winmux's default PANE/WINDOW/SESSION `display-menu` context menu (SP7, closes #51). SP7 makes most of the above real, rebindable `Bindings` table entries (`bind-key -T root MouseDown1Pane <command>` now works, closing #57/#67) instead of hardcoded dispatch, and adds **application mouse passthrough** (closes #35/#72): a pane whose OWN app has requested mouse reporting gets its clicks/drags/wheel re-encoded and forwarded to it instead of consumed for winmux's own copy-mode/drag handling — see the Hard-won platform gotchas entry below for why this can't be proven with a real hosted app over Windows ConPTY in this project's e2e harness (unit-level coverage only).

Prompt/confirm input is captured at the server layer (a line editor: printable append, BSpace delete, Enter commit, Esc/Ctrl-c/Ctrl-g cancel) so prompt keystrokes are never misinterpreted as bindings.

### escape-time and automatic-rename (sub-project 4)

`escape-time` (default 500ms, `set -g escape-time <ms>`) governs how long the server waits after a lone `Escape` byte before treating it as a real Escape keypress rather than the start of an Alt/CSI sequence still in flight — driven by the server's 50ms `Tick` handler flushing any client whose pending decode buffer has aged past the configured window. `automatic-rename` (default `on`) renames a window to the first token of its active pane's ConPTY OSC-0/2 title whenever that title changes (throttled to once per window per 500ms); any manual `rename-window` (including the `,` prompt) permanently disables automatic renaming for that window until reversed by config/another explicit rename call. `allow-rename` (SP7, default `off`, window-scoped) is a SEPARATE, independent gate on the legacy `ESC k <name> ESC \` title-rename escape only — unlike `automatic-rename`, it is never blocked by prior manual renames and doesn't read `automatic-rename` at all (matches real tmux's `input_exit_rename` exactly); `Grid::feed` pre-scans/strips this escape out of the VT byte stream before `vte::Parser` ever sees it (no string-capture path exists for it in the `vte` crate), including across back-to-back `ESC k` sequences with no intervening terminator — relevant because real Windows conhost silently drops the literal `ESC \` terminator on a hosted process's output (see "Hard-won platform gotchas" below), making unterminated back-to-back `ESC k` the realistic pattern on this platform, not an edge case.

### Option scopes (SP7)

Real per-session and per-window option overlays (`options::Overlay`, closing follow-ups #26/#32/#68): unprefixed `set` from an attached client targets that client's own SESSION; `setw`/`set -w` targets its current WINDOW; `-g` always targets the GLOBAL level regardless of scope. Reads resolve local-override-else-global-default via `_for`-suffixed getters (`Options::pane_base_index_for`, `status_style_for`, etc.), keyed by a table-decides-scope classifier (`options::scope`, per `docs/tmux-reference/commands-config-options-formats.md` §3.3.4 / Appendix B) — most options are Server(global)-only or Session-scoped; only a handful (`pane-base-index`, `automatic-rename`, `allow-rename`, `main-pane-width`/`-height`, `monitor-bell`/`-silence`, `window-status-*`, `clock-mode-*`, `word-separators`, `pane-border-indicators`) are Window-scoped. **Headless calls (CLI/config loading) with no acting session/window always fall back to the global table** — this is the pre-existing, unchanged behavior for every caller that predates SP7, and it means a CONFIG-file `setw <opt> <val>` (no acting window exists yet at load time) becomes a GLOBAL default, not a per-window override, even though the identical command typed at the `:` prompt (which DOES have an acting client/window) becomes a true local override — `tests/e2e_config.rs::config_sp7_surface_takes_effect` demonstrates both halves of this distinction end to end. Documented narrowings (see follow-ups #75-#77): no `-t` targeting for `set`/`show` (always the acting client's own session/window), no per-PANE option tree, live `set` SIDE EFFECTS (prefix rebinding, mouse SGR enable/disable, status relayout) still fire on GLOBAL writes only even though the VALUE itself is scoped correctly, and `status`/`status-position` pane-area GEOMETRY stays global-only (the row-content options built from them are per-entity already).

### Alerts (SP7)

Real bell/activity/silence monitoring (closes follow-up #74), matching tmux's split between a MONITORING gate and a separate REACTION gate: `monitor-bell` (window, default **ON** — the opposite default from the other four alert options below), `monitor-activity` (window, default off), `monitor-silence` (window, seconds, default 0/off) each independently control whether `model::Window::mark_bell`/`mark_activity`/`mark_silence` sets a window flag at all (`#`/`!`/`~` in `window_flags`, tmux's fixed order); `bell-action`/`activity-action`/`silence-action` (session, `any`/`none`/`current`/`other`, defaults `any`/`other`/`other`) then separately gate whether that same event ALSO triggers a visual reaction (`visual-bell`/`visual-activity`/`visual-silence`, session, off by default) or the terminal BEL passthrough. `window-status-bell-style`/`-activity-style` (window) layer over a flagged tab's base style, bell beats activity/silence. Flags clear on visiting the window — wired into every real current-window-changing call site, not just the obvious ones (see `model::Session::clear_alerts_for`'s doc comment for the full list, a discovery made mid-implementation: several `server/dispatch.rs` call sites mutate `session.current` directly rather than going through `model.rs`'s own setter methods).

### Config files

With no `-f`, the server (at startup) tries `%USERPROFILE%\.tmux.conf` (or `$XDG_CONFIG_HOME/tmux/tmux.conf` if that env var is set) first, then `%USERPROFILE%\.winmux.conf` second — both optional, missing files are silently skipped, later files can override earlier ones. `-f <file>` (top-level flag, e.g. `winmux -L mysock -f myconf.tmux.conf`) replaces that default chain with exactly the file(s) given (required — a missing explicit file is a startup error); `-f -` disables config loading entirely (tmux's `-f /dev/null` idiom). `-f` only has an effect on the invocation that actually autostarts the server — tmux semantics: config is read once at server start, so `-f` against an already-running server is a no-op.
